新人指導方針メモ

基本スタンス

  • 小言は言わない。注意が必要なときだけ注意する
  • 「お父さんになったつもりで」くらいの距離感でちょうどいい
  • 業務上の問題（締め切りなど）以外はあまり口を出さない
  • 答えを教えるより、考え方の手順を見せる

導入の進め方

ステップ1：本人のコードを見る

  • 卒業制作のゲーム（DXLibで制作）のソースを見せてもらう
  • 「なんでこう書いたの？」を詰問ではなく興味として聞く
  • 本人が自分のコードを説明できるか確認する

ステップ2：改造課題

  • 「このゲームに別の機能を追加するにはどうするか」を考えさせる
  • 正解がない問題なので、思考の筋道を作る練習になる
  • 改造の過程を言語化させる

ステップ3：仕事の文脈に繋げる

  • 本人の土台から「うちの現場ではこういう制約がある」と橋渡しする
  • いきなり業務コードを渡さない
  • ペアプロで、作業しながら「なぜこうするか」を声に出して見せる

教えるべき考え方

「なぜそうなるのか」を常に考える習慣

  • 出向が多い現場では、知らない環境で自分で調べて動かす力が必須
  • 「教えてもらえなかったから教えられない」と言う人間にならないように
  • 調べ方と、なぜ調べるのかを自分で示す

データ構造と処理を先に整理する

  • 「手を抜くために先に苦労する」
  • 最初に設計を整理しておくと、あとで「コピーして名前置換するだけ」になる
  • 迷いながら書いたコードは制御フローに迷いが出る
  • 上から下にきれいに流れるコードが書けるかどうかは、頭の整理と直結している

引き出しを増やす意識

  • 過去ソースから自分のライブラリを作っていく
  • 一度書いたロジックを別の現場でも使えるようにする
  • コードを書く速度はあとでどうにでもなる

「わからないことがわからない状態」

  • 情報量が少なすぎて比較ができない状態
  • 知識がないから、何が欠けてるかも見えない
  • わからなくていいから「動かす」を先にやらせる
  • 「こうしたいけどできない」と思わせる訓練
  • 「わからないんです」ではなく、状況を具体化させる
  • 「今どこで詰まってる？」ではなく「今何をしようとしてる？」と聞く

AIの使い方ルール

Claudeへの質問はOK

  • わからないことを聞くのは問題なし
  • ただし「どうしてもわからないところを聞く」であって、「最初からAIに投げる」は禁止

業務コードへのAI生成コードの混入は禁止

  • 品質保証・著作権・説明責任の観点から

「説明できないコードは採用しない」ルール

  • AIでも自分のコードでも同じ基準
  • 説明できなければ採用しない
  • コメントを書けるかが目安

正しい使い方の順番

  1. 自分で考えて書く
  2. レビューをもらう
  3. 理解して修正する
  4. 答え合わせとして参考にする

習熟の見込み

  • 即戦力は期待しない
  • 最短でも1年半が現実的な見積もり
  • ゲーム系専門卒でも業務C環境は別物
  • 焦らせると長期的に損

最初に入れる現場について

  • 今の現場には入れない
  • 別メーカー案件から経験させる
  • まともな現場で判断力を育てる
