### 前提
#### 対象
- ゲーム系専門学校の卒業生
- CやC++で卒業制作
- 校内でUnityなどを使ってチーム制作も経験済み

#### 流れ
1. 現状は卒業制作をセルフレビュー中
	- ただ、何書いて良いかわからないみたいで結構文章量はすくない
2. そのレビュー内容を確認しながら「その内容は他人に伝わる内容か？」確認させる
	- 自分が書いた内容を他人に説明することは必要だから
3. 自分が直したいと感じた、直そうと思うところをリストアップ
	- なぜそう思ったのかを明示させる
4. 実際に直させる
	- 直した内容はメモしてまとめる
	- 本当にその修正でいいのかは考えさせる
5. 直した内容を説明してもらう
	- 必要な修正だったのかも確認する
6. 全面的に書き直させる
	- どういう作り方か確認して、タスクリスト形式で書かせる

#### タスクリスト形式
Cで書く場合。
```
struct tagBaseObj {
	struct tagBaseObj *prev;
	struct tagBaseObj *next;
	struct tagBaseObj *Sibling;
	unsigned short action;			// 各ビットでフラグにする
	void (*init)( struct tagBaseObj * );		// 自分自身へのポインタ
	void (*update)( struct tagBaseObj *);
	void (*draw)( struct tagBaseObj *);
};

typedef tagBaseObj BASEOBJ;
```

C#とかC++ならクラスに落とし込む

```
struct BaseObj {
    BaseObj *prev;
    BaseObj *next;
    BaseObj *sibling;
    unsigned short action;

    BaseObj() : prev(nullptr), next(nullptr), sibling(nullptr), action(0) {}
    virtual ~BaseObj() {}

    virtual void init()   {}
    virtual void update() {}
    virtual void draw()   {}
};
```

### 注意させること
- コメントなるべく書く
	- 内容によってはソースだけでいいこともあるが、面倒な処理は書いておくといい
		- 3日後の自分は他人
- マジックナンバーを使わない
	- 数字には意味があるはずなので、その意味を記載する
		- 使う直前で定義してもいいが、大体は先頭にまとめる
		- そうすることで変更が楽になる。(コメントは忘れずに)
- 変数名はわかりやすい名前にする
	- 長ければいいというものではない
		- ControllerpreviousFrameKeyBuf 
		- 例えば、 prevInputState 以前の入力状況って意味になるのでよりわかりやすくなる
		- previousと書いてもいいけど、省略形prevで意味は伝わる
- 型とサイズの話
	- 型とサイズの話（sizeof使え、charとintは別物）
- if ～ return true/falseで書くなら条件式の結果を返すだけでよい
	- それそのものが論理式だから。(条件式とか)
	- TRUE/FALSEが結果になる式だから。
- スコープの話
	- 変数の有効性の問題
		- グローバル
		- スタティック
	- ファイル内だけで使う変数ならば、なるべくstatic
		- 外部から値を参照する場合はgetterを作るなど
		- C++とかC#っぽいアプローチでCで書くならなおさら

### しつもんな
- 一通り目を通していますが、書くことないとなっているものは「なぜそうしているのか」が書いてあるとわかりやすくていいですね
	- 定義をしているから、という理由でもいいのですが、なぜそういう風にしたのかがあるといいと思います
	- この辺は、自分自身に対して「なぜ？」とか「どうして？」「どこが？」「いつ？」「どっちが？」「なにが？」の5W1Hで質問しながら見直しと修正するのがいいんじゃないでしょうか。

- 元を書いたのは自分だけど、今はその当時の自分ではないので思い出しながら調べて行くのがいいと思います。

- それから、全体的にベースはC++で書いてあるように見えたんですが、いくつかはC++とはちょっとちがう書式になっているソースがあったのでそういった点もなぜそうしてるのか記載があるといいですね

そうなんです、プログラムを書くのも大変ですが、文章書くのも意外と大変なんです。

プログラムを見てもらえば、流れはわかるだろうというのは確かにそうなんですが、
人に説明するとなると、どう順序立てていけばいいかとか迷いますよね。
ただ、このプログラムを見て、ほかの人が気になるとしたらどこだろうみたいなことも
考えられるようになると、自分が知りたいこと、わからないところなんかも見えてくる
ようになってくると思います。

それとある程度資料を作れるようにしておくと、この先プログラムの自分なりに人に
説明するためのコツなんかもつかめるようになります。
(結局自分だけで終わる仕事ってあんまりないので人に説明するのは必要なスキルの1つです)

bullet.md に追加で記載された説明で未使用変数を確保している理由や現状未使用になっていることを記載しているのはよかったと思います。
この手の話は、コメントなどで書くことも必要なんですが、メモ書きなどが残っていると、よりその意図をくみやすくなります。

説明できれば、残しておくか一旦コメントアウトするかなども時間が経ってからも判断付きやすいので。

C++とCのstaticの扱いってどうやって説明すればいいかね

C++の場合、変数はそのクラスからしか参照できないメンバー、関数も同じようなイメージ(privateとかあったっけ)
Cの場合はそのソースの中、関数の中からしか参照できない外に出さない変数/関数ってイメージなんだけど

18:51

Differentiated static usage across C and C++ contexts comprehensively
Differentiated static usage across C and C++ contexts comprehensively
大体合ってる。整理すると：

Cのstatic

グローバル変数/関数につけると「このファイル内だけ」（外部リンケージを切る）
ローカル変数につけると「関数抜けても値が残る」
C++のstatic（クラスメンバー）

変数：インスタンスじゃなくクラス全体で1個だけ持つ
関数：インスタンスなしで呼べる
privateはアクセス制御で別の話で、staticとは直交する概念だから混ぜない方がいいかも。

新人への説明としては「Cのstaticはスコープを狭める、C++のstaticはクラスに縛り付ける」って切り分けると混乱しにくいかな。
